# Brown Biotech Local Operator SOP

## Purpose
Use local tools consistently so every request becomes a brief, a route, and a deliverable.

## Default tool choices
- **Notion** → source of truth, HQ, weekly review, decision log
- **Open Design** → local control layer for launching the right agent
- **Claude Code** → larger implementation tasks and repo edits
- **Codex** → scoped implementation, refactors, cleanup, quick fixes
- **Telegram** → approvals, escalations, human confirmation

## SOP
### Step 1 — Read the request
- identify the lane
- identify the risk level
- identify whether approval is needed

### Step 2 — Choose the tool
- use Claude Code for larger changes
- use Codex for fast scoped changes
- use Open Design when you need a local launcher or agent picker
- use Telegram when the task needs human confirmation

### Step 3 — Execute the work
- keep the request scoped
- keep the language consistent with HQ
- write the result back into the operating system

### Step 4 — Publish the receipt
- record route
- record owner
- record next action
- record approval state if relevant

### Step 5 — Update HQ
- update Notion
- update weekly review if needed
- convert the work into a reusable template if it repeats

## Escalation rules
Escalate to human approval for:
- money
- contracts
- deployment
- public statements
- medical or clinical claims

## When to use Open Design
Use Open Design when you want:
- a local launcher for multiple agents
- a visible operator surface
- a quick way to choose Claude Code, Codex, or another tool
- a desktop control center for work instead of a chat-only flow

## When to use a visual board
Use a visual board when the team needs to see:
- intake flow
- lane state
- proof state
- approval state
- weekly status

## Completion standard
A task is done only when the work is recorded back into the system and the next owner can pick it up without explanation.
